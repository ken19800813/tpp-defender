### 2.1 架構決策記錄

- **拆分 main.py / bot_engine.py**：bot_engine 負責「監控 + 偵測 + 傳送」的無 UI 邏輯，main.py 負責 UI 事件迴圈跟彈窗管理，便於測試和重用。
- **60 秒輪詢不改 WebSocket**：Render 免費層 sync worker 無法承載千人級持久連線，縮短輪詢到 15-20 秒已經可達「接近即時」，Turso 成本仍在免費額度內，收益遠大於升級付費方案的成本。
- **廣告透過 HTTP URL 不存本地**：節省磁碟空間、避免版本不同步；UI 層負責下載 + PIL 縮放到固定寬度。

## 3. 核心功能邏輯

### 3.1 YouTube 聊天監控（bot_engine.py）

```python
YouTubeLiveTacticalBot:
  - start_monitor()：等待聊天區建立 → 持續監控新訊息
  - _detect_message()：檢查是否觸發關鍵字規則
  - _try_send(text)：全域冷卻閘 + 單次彈窗限制 → 傳送
  - _send_chat_message(text)：透過 Playwright 自動輸入+傳送
```

**流程**：
1. 使用者登入已登帳號、開啟直播
2. 監控迴圈每 0.5 秒檢查一次聊天區 DOM
3. 新訊息進來 → 檢查黑名單（channel name）→ 檢查關鍵字
4. 命中規則 → UI callback（`ALERT` 或 `AUTO_SENT`） → 主執行緒彈窗/通知
5. 手動按「送出」或自動模式 → `_try_send()` → 冷卻檢查 → Playwright 輸入+傳送

### 3.2 反擊彈窗（NotificationPopUp）

- **可編輯**：使用者可在彈窗內改反擊文字
- **髒話檢查**：整句 `find_forbidden_word()` 檢查，有髒話則「送出」按鈕 disabled
- **單次限制**：`_has_sent = True` 後按鈕 disabled，強制關閉重開才能送第二次
- **冷卻倒數**：如果還在 10 秒冷卻期內，按鈕改顯示「冷卻 3 秒」等倒數數字

### 3.3 全自動模式（auto_send_enabled）

- 規則命中 → 自動選優先規則（如果有）或命中規則的回覆 → 加 @mention → 傳送
- **同時彈通知窗**（`AutoSentNotice`）：不是無聲，使用者知道發生了什麼
- **仍受冷卻約束**：不管多少條規則同時命中、對方發多少則訊息，一律共享 10 秒冷卻
- **不能鬼祟**：明確提示「系統已自動回覆 @xxx」，杜絕被說是人工洗版

### 3.4 廣告推播（60 秒輪詢）

```python
_check_ads():               # 每 60 秒執行一次
  ↓
_check_ads_worker():        # 背景執行緒
  ↓
config_mgr.fetch_new_ads(): # GET /api/livestream/ads/active
  ↓
filter([ad for ad in ads if ad['id'] not in self.config.seen_ad_ids])
  ↓
download image + mark as seen → AdPopUp(image_bytes, title, link)
```

**AdPopUp 特性**：
- 固定寬度 380px（彈窗內容區）
- 高度依圖片比例自動增高，不裁切
- 手動「關閉」按鈕 + 超連結按鈕
- 同一台電腦同一支廣告只顯示一次（`seen_ads.json` 本機記錄）

## 4. 這個工作階段做了什麼（2026-07-14）

### 4.1 廣告彈窗 UI 調整

**改動 1**：移除高度裁切上限（原本 320px）
- 原因：用戶提交的廣告圖片可能是直圖（高寬比 3:2）或其他比例，裁切會扭曲內容
- 修法：`_resize_to_fixed_width()` 改為只固定寬度 380px，高度依圖片比例自動縮放
- 結果：彈窗高度會隨圖片浮動，排版會自適應（CustomTkinter frame 自動調整）

**改動 2**：確認圖片來源支援

用戶問：「是否支援 YT / FB / Google Drive / 搜圖等來源？」

答案：**只要是 HTTP GET 能直接返回圖片位元資料的 URL 都行**，但各平台陷阱不同：
- **YouTube 縮圖**：用 `https://i.ytimg.com/vi/{影片ID}/hqdefault.jpg` 格式
- **Facebook**：右鍵「複製圖片連結」拿到 `scontent.xx.fbcdn.net/...`，但常有簽章過期問題
- **Google Drive**：改用 `https://drive.google.com/uc?export=view&id={檔案ID}` 格式，檔案必須「知道連結的人都可檢視」
- **其他**：直接圖片 URL，看來源網站是否允許盜連

**建議**：未來廣告量增加時，改為在後台上傳圖片到 LINEBOT 的 `/static/images/` 資料夾，確保穩定性、避免第三方平台的防盜連機制。目前不是阻塞點，但這是良好的長期方向。

### 4.2 輪詢架構評估與決策

**用戶提問**：「改成 15 或 20 秒輪詢會怎樣？」

**評估結果**：
| 指標 | 現況 60s | 改 20s | 改 15s |
|------|---------|--------|--------|
| 每秒請求（千人） | ~17 | ~50 | ~67 |
| Sync worker 壓力 | ✅ 零 | ✅ 零 | ✅ 零 |
| Turso row reads/月 | ~1-2 億 | ~3-6 億 | ~4-8 億 |
| 免費額度用量 | 10-20% | 30-60% | 40-80% |
| 廣告延遲 | 最多 60s | 最多 20s | 最多 15s |
| 額外成本 | $0 | $0 | $0 |

**決策**：使用者選擇**維持 60 秒輪詢**，不做改造。理由是「推播及時性不是關鍵需求，現有架構已夠用」。

## 5. 重大技術決策記錄

### 5.1 廣告推播架構選擇

**背景**：使用者預期「數百到數千人同時使用」。出現兩個方案競爭：

**方案 A：WebSocket/SSE 主動推播**
- 優點：毫秒級延遲，接近即時
- 缺點：Render 免費層 sync worker 無法承載千人級持久連線（worker 數有限，每個連線卡死一個 worker）
- 成本：需升級付費方案（最低 $12/月）+ 改整個後端架構（換 async worker）
- 風險：會衝擊 LINE Bot 的穩定性（webhook 會被排隊/逾時）
- **判定**：現階段不可行

**方案 B：縮短輪詢間隔（15-20 秒）**
- 優點：無架構改造、Turso 成本仍在免費額度內
- 缺點：延遲最多 15-20 秒（非毫秒級）
- 成本：$0（帶寬包含在 Render 免費層月費內）
- 風險：無
- **判定**：推薦，但使用者選擇維持 60 秒

**最終決策**：保留 60 秒輪詢，架構不動。理由：「推播即時性不是應用場景的關鍵需求」。

**常見問題**：

Q: Push 時收到「refusing to allow a Personal Access Token to create or update workflow」
A: Token 權限不足（缺 `workflow` scope）。重新生成一個新 token（Settings → Developer settings → Personal access tokens），勾選 `repo` + `workflow` + `admin:repo_hook` scope。

Q: GitHub Actions 編譯失敗
A: 檢查 https://github.com/ken19800813/tpp-defender/actions 看詳細錯誤。常見原因：
- 依賴缺失：檢查 `requirements.txt` 是否更新
- Python 版本不符：Workflow 用 Python 3.11，本機版本要一致
- 打包檔案不存在：spec 檔路徑確認無誤

Q: Mac .app 或 Windows .exe 打包大小太大
A: PyInstaller 預設會包所有依賴。`TPPchat.spec` 的 `excludes` 參數可指定要排除的模組（例如測試框架），但目前配置已是最小化。

## 附錄：廣告彈窗圖片尺寸建議

若使用者在後台上傳廣告圖片，建議尺寸（考慮 380px 固定寬度的縮放）：
- 寬度：380px 或倍數（縮放會自動 LI NCOS 重取樣）
- 高度：200-400px 較佳（彈窗不會太小也不會太高）
- 縱橫比：16:9 或 1:1 最常用
- 檔案格式：JPG（最小化檔案大小，60s 輪詢時 bandwidth 關鍵）

例：廣告圖片上傳時 1920x1080 JPEG，彈窗會自動縮到 380x214，檔案也會被 PIL 壓縮。
